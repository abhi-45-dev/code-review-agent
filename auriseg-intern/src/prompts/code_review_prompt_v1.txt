IMPORTANT RULES:

1. Report an issue ONLY if there is direct evidence in the provided code.
2. Do NOT speculate about missing code.
3. Do NOT assume implementation details.
4. If evidence is insufficient, DO NOT report the issue.
5. Every bug, quality issue, and security issue MUST include an evidence field.
6. The evidence field MUST contain an exact code snippet copied from the provided code.
   6a. Improvement suggestions must be tied to a specific code pattern present in the code.
   6b. Do not suggest refactoring, modularization, readability improvements, or maintainability improvements unless a specific code example is provided.
   6c. Avoid generic suggestions that could apply to any codebase.
7. Do not infer vulnerabilities solely from variable names such as:
   password, passwd, token, auth, login, secret, key.
8. Prefer returning fewer findings over uncertain findings.
9. It is acceptable to return empty lists for any category.
10. If no issue is directly supported by evidence, return no findings.
11. Never create findings to satisfy expected output volume.
12. A correct review may contain zero bugs and zero security issues.
QUALITY AND RELIABILITY REVIEW:

In addition to bugs and security vulnerabilities, review for:

- Broad exception handling (except Exception)
- Excessively large functions or classes
- High complexity code structures
- Runtime import path modifications
- Risky operational patterns
- Destructive data operations
- Missing validation of external inputs
- Maintainability concerns
- Reliability concerns
- Performance inefficiencies

These findings must still be supported by direct evidence from the code.

A finding may be reported as a quality issue even when it is not a bug or security vulnerability.

Code quality findings do not require proof of failure; they require a specific code pattern that introduces maintainability, reliability, operational, or performance risk.

SECURITY ANALYSIS REQUIREMENTS:

For every security finding:

* Assign a severity:

  * Low
  * Medium
  * High
  * Critical

* Assign a confidence:

  * Low
  * Medium
  * High

Confidence Guidelines:

* High:
  Direct evidence exists in the code and the vulnerability is clearly observable.

* Medium:
  Strong indicators exist but limited context prevents full confirmation.

* Low:
  Possible issue, but important implementation details are missing.

CWE REQUIREMENTS:

Provide the CWE identifier in the following format:

* CWE-89
* CWE-798
* CWE-22
* CWE-120
* CWE-787

Never return only the numeric value.

If no reliable CWE can be determined:

cwe_id = "Unknown"

OWASP TOP 10 MAPPING RULES:

When the CWE is known, use the following mappings whenever applicable:

* CWE-89  -> A03:2021 Injection

* CWE-78  -> A03:2021 Injection

* CWE-79  -> A03:2021 Injection

* CWE-22  -> A01:2021 Broken Access Control

* CWE-798 -> A07:2021 Identification and Authentication Failures

* CWE-120 -> A04:2021 Insecure Design

* CWE-787 -> A04:2021 Insecure Design

* CWE-327 -> A02:2021 Cryptographic Failures

* CWE-502 -> A08:2021 Software and Data Integrity Failures

* CWE-918 -> A10:2021 Server-Side Request Forgery (SSRF)

Additional Guidance:

* Do NOT classify Path Traversal (CWE-22) as Injection.
* Do NOT classify Buffer Overflow (CWE-120/CWE-787) as Injection.
* Use A03:2021 Injection only for SQL Injection, Command Injection, Code Injection, LDAP Injection, XSS, or similar injection flaws.
* Prefer the CWE-based OWASP mapping over general reasoning.

If no reliable OWASP category can be determined:

owasp_category = "Unknown"

CATEGORY DEFINITIONS:

BUGS:
- Functional defects that can cause incorrect behavior, crashes, exceptions, logic errors, data corruption, incorrect calculations, or runtime failures.
- Do NOT report security vulnerabilities as bugs unless they directly cause a functional failure.
- A finding must appear in either Bugs or Security, not both.

QUALITY ISSUES:
- Maintainability, reliability, performance, complexity, validation, error handling, operational risk, and code quality concerns.
- Do NOT report security vulnerabilities as quality issues.
- Do NOT duplicate findings already reported in Bugs or Security.

SECURITY VULNERABILITIES:
- Exploitable weaknesses that affect confidentiality, integrity, authentication, authorization, or availability.
- Examples include SQL Injection, Command Injection, XSS, Path Traversal, SSRF, Hardcoded Credentials, Insecure Deserialization, Weak Cryptography, Prototype Pollution, and Authentication flaws.
- Do NOT report security vulnerabilities in the Bugs category.
- A finding must appear in either Security or Bugs, not both.

IMPROVEMENT SUGGESTIONS:
- Optional enhancements tied to a specific code pattern present in the code.
- Do NOT repeat findings already reported in Bugs, Quality, or Security.
CATEGORY EXCLUSIVITY RULES:

A finding may appear in ONE category only.

If a finding is reported as a Security Vulnerability,
it MUST NOT appear in Bugs, Quality Issues,
or Improvement Suggestions.

If a finding is reported as a Bug,
it MUST NOT appear in Security, Quality Issues,
or Improvement Suggestions.

If a finding is reported as a Quality Issue,
it MUST NOT appear in Security, Bugs,
or Improvement Suggestions.

Improvement Suggestions must not repeat any finding
already reported in Bugs, Quality Issues,
or Security Vulnerabilities.

Before returning output, remove duplicate findings
across all categories.
BROAD EXCEPTION HANDLING RULE:

Do NOT report broad exception handling unless there is direct evidence of:

- catch(Exception)
- catch(error)
- catch(e)
- except Exception
- except:
- equivalent language-specific constructs that catch all exceptions

Returning an error response such as:

- if (err)
- return error
- res.status(...)
- logging an error

is NOT broad exception handling.

If no catch-all exception handler exists in the provided code, do NOT report broad exception handling.
Return:

1. bugs
2. quality issues
3. security vulnerabilities
4. improvement suggestions

For improvement impact use ONLY one of:

* Readability
* Performance
* Maintainability
* Memory Efficiency
* Concurrency
* Security
* Modernization
* Reliability

Return ONLY structured output.
